Part 1 · How a web app works

The 10-second picture

[ Browser ] ──HTTP request──▶ [ Server ] ──SQL──▶ [ Database ] │ [ Browser ] ◀──JSON response── [ Server ] ◀──rows──

Every click, search, and form submit boils down to this. The browser asks. The server answers. The database stores state.

Part 1 · How a web app works

A real flow: sending a mentorship request

1. User fills form, clicks "Send" 2. Browser sends POST /api/mentorship/request with form data as JSON 3. Server receives it — runs your code 4. Server checks: is this mentor open? at capacity? who's asking? 5. Server writes a row to Postgres: mentorship_request 6. Server emails the mentor via Resend 7. Server responds: { ok: true, requestId: "abc" } 8. Browser shows a toast: "Request sent"

Steps 3–6 are where all the business rules live. That's the code we need to get right.

Part 2 · TypeScript, briefly

Types describe the shape of your data

TypeScript is JavaScript + a type system. You tell the compiler what shape data must have, and it catches bugs before users do.

type Mentor = {
  id: string
  name: string
  mentor_open: boolean
  max_pending: number
}

// The compiler errors if you try to do this:
const m: Mentor = { id: "abc", name: "Rick" }
//                  ^^^^^^^^^^^^^^^^^^^^^^^^^
// missing: mentor_open, max_pending

Types are erased at build time. They don't exist at runtime — they're a safety net for you, not a library.

Part 2 · TypeScript, briefly

Functions have input types and output types

function greet(name: string): string {
  return `Hello, ${name}`
}

// Input: one string. Output: one string.
// Calling greet(123) is a compile error.

Later we'll see a function whose return type is a union — it returns either a success shape or a failure shape, and TypeScript forces the caller to handle both.

type Result =
  | { ok: true; value: number }
  | { ok: false; error: string }

// The caller must check `result.ok` before touching `value` or `error`.
Part 3 · Next.js API routes

An API route is a file that handles HTTP

In Next.js App Router, the file's path becomes the URL.

// File: src/app/api/mentorship/route.ts
// URL:  POST /api/mentorship

export async function POST(req: Request) {
  const body = await req.json()       // read incoming JSON
  console.log('got body:', body)
  return Response.json({ ok: true })  // send JSON back
}

req is the incoming HTTP request. Whatever the function returns is sent back to the browser.

Part 4 · The problem

What "business logic in the route" looks like

// src/app/api/mentorship/route.ts
export async function POST(req: Request) {
  const body = await req.json()
  const session = await getServerSession()
  if (!session) return Response.json({ error: 'unauth' }, { status: 401 })

  // ⚠️ Business rules start here, tangled with HTTP plumbing:
  const mentor = await supabase.from('profiles')
    .select().eq('id', body.mentorId).single()

  if (!mentor.data?.mentor_open)
    return Response.json({ error: 'closed' }, { status: 400 })

  const pending = await supabase.from('mentorship_request')
    .select('*', { count: 'exact' })
    .eq('mentor_id', body.mentorId).eq('status', 'pending')

  if (pending.count >= mentor.data.max_pending)
    return Response.json({ error: 'full' }, { status: 400 })

  const request = await supabase.from('mentorship_request')
    .insert({ ...body, status: 'pending' }).select().single()

  await resend.emails.send({ to: mentor.data.email, /* ... */ })
  return Response.json(request.data)
}
Part 4 · The problem

Nothing wrong with this today. Tomorrow it breaks.

  • Mobile app needs the same rule — you copy-paste the logic into another endpoint.
  • Weekly worker auto-pauses inactive mentors — it needs "is mentor open?" — copy-paste.
  • Admin panel wants to create a request on behalf of someone — copy-paste.
  • Unit test requires mocking Request, Response, Supabase, Resend — painful, so you skip it.
  • Bug found in the rule — you fix it in one place, forget the other four copies.

The rule "mentor must be open and under capacity" is a business concept. It shouldn't live inside a web framework's request/response handler.

Part 5 · The /lib pattern

Split into two files

/lib/mentorship/createRequest.ts

Pure business logic. No HTTP. No Next.js. Framework-agnostic.

/app/api/mentorship/route.ts

Thin transport wrapper. ~10 lines. Parses input, calls lib, returns response.

The lib function can be called from an API route, a server action, a background worker, a test, or a completely different service in the future.

Part 5 · The /lib pattern

Define input and result types first

// src/lib/mentorship/createRequest.ts

export type CreateRequestInput = {
  menteeId: string
  mentorId: string
  reason: string
  helpNeeded: string
}

export type CreateRequestResult =
  | { ok: true;  requestId: string }
  | { ok: false; error: 'mentor_closed' | 'mentor_full' | 'not_found' }

CreateRequestResult is a union: either success with a requestId, or failure with a specific error. The compiler forces any caller to handle both branches.

Part 5 · The /lib pattern

The function — business rules, nothing else

export async function createMentorshipRequest(
  input: CreateRequestInput,
  deps: { db: DbClient; notify: Notifier }
): Promise<CreateRequestResult> {

  const mentor = await deps.db.profiles.findById(input.mentorId)
  if (!mentor)               return { ok: false, error: 'not_found' }
  if (!mentor.mentor_open)   return { ok: false, error: 'mentor_closed' }

  const pending = await deps.db.mentorshipRequests.countPending(input.mentorId)
  if (pending >= mentor.max_pending)
    return { ok: false, error: 'mentor_full' }

  const request = await deps.db.mentorshipRequests.create({
    ...input, status: 'pending',
  })
  await deps.notify.mentorshipRequestCreated(mentor.email, request)
  return { ok: true, requestId: request.id }
}

Notice: no Request, no Response, no Next.js imports. deps is how we pass in the database and notifier — injected instead of imported, so we can swap them for testing.

Part 5 · The /lib pattern

The route handler becomes trivial

// src/app/api/mentorship/route.ts
import { createMentorshipRequest } from '@/lib/mentorship/createRequest'
import { db } from '@/db/server'
import { notify } from '@/notify/resend'
import { requireSession } from '@/lib/auth/session'
import { createRequestSchema } from '@/lib/mentorship/schemas'

export async function POST(req: Request) {
  const session = await requireSession()
  const input = createRequestSchema.parse(await req.json())

  const result = await createMentorshipRequest(
    { ...input, menteeId: session.userId },
    { db, notify }
  )

  if (!result.ok)
    return Response.json({ error: result.error }, { status: 400 })
  return Response.json({ requestId: result.requestId })
}

Parse input. Load session. Call lib. Map result to HTTP. That's the whole handler.

Part 6 · Why this future-proofs BridgeCircle

One function, many callers

/lib/mentorship/createRequest.ts ▲ │ ┌───────────────────────┬───────┼───────────────────────┐ │ │ │ │ [API route] [server action] [Inngest worker] [test] web user clicks form mutation weekly sweep auto- fake db, "Send request" from a page pauses mentors instant check

Bug fix in one place. Rule change in one place. New caller tomorrow? It just imports the function.

Part 6 · Why this future-proofs BridgeCircle

When the native mobile app arrives

The mobile app needs the exact same rule: "is mentor open? under capacity? create a request, notify them."

  • Option A: mobile app calls the same /api/mentorship endpoint you already built. Zero changes.
  • Option B: you want a dedicated backend service for mobile. Copy /lib/mentorship/ into it. No Next.js tangles to unwind.

Either way, the rules were never coupled to the web framework. That's the payoff.

Part 7 · How it looks in the repo

Folder layout

app/src/ ├── app/ ← Next.js routes (HTTP + UI) │ ├── api/ POST/GET handlers (thin wrappers) │ └── (member)/ pages, server components ├── components/ui/ shadcn components ├── lib/ ← business logic (framework-free) │ ├── mentorship/ createRequest, acceptRequest, autoPause │ ├── profile/ update, computeCompleteness, applyVisibility │ ├── search/ rank, applyFilters, extractEntities │ ├── invite/ validate, send, resend, expire │ └── events/ createEvent, rsvp ├── db/ ← typed Supabase wrappers └── notify/ ← Resend + push wrappers

The rule of thumb: anything you'd explain to a non-engineer as a product rule lives in /lib/.

Part 8 · The rule to remember

Route handlers do four things. Nothing else.

  1. Parse input (zod)
  2. Load session / check auth
  3. Call a /lib function with dependencies
  4. Map the result to an HTTP response

If a line of code looks like a business rule ("mentor must be open," "invite must be unexpired," "user must be friends to DM") and it's inside /app/api/, it's in the wrong place. Move it to /lib.

That's it.

One folder, one rule, and BridgeCircle stays portable through mobile, scale, framework churn, and future backends.
