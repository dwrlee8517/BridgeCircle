04 / surface rules

Apply states differently by product surface.

These are the recurring BridgeCircle patterns where state treatment will either help or damage the experience.

Surface Required state behavior What to inspect
Person cards Hover, focus, selected, paused, and full mentor states preserve height. Primary action is visible without browsing.
Request cards Incoming pending requests show an obvious primary action. Outgoing pending is quieter. Owner of the next action is clear.
Inbox rows Unread uses dot plus weight. Selected uses primary tint without movement. Read/unread and selected can coexist.
Event RSVP Going, waitlisted, not-going, and pending preserve button footprints. RSVP state does not shift surrounding copy.
Admin tables Selected and bulk rows use primary tint. Revoked rows stay visible with muted text. Dense state still meets contrast expectations.
Forms Disabled and pending buttons preserve width. Errors stay attached to fields. Mobile form errors remain adjacent to the field.
Notifications Unread uses dot plus weight. Action-needed uses warning tint. No arbitrary accent colors for notification categories.
Email No app motion. State comes from copy, one CTA, and email-safe tokens. Plain-text fallback preserves the action.

05 / QA checklist

Use this when reviewing real screens.

A screen passes only when state, hierarchy, action clarity, accessibility, and responsive behavior agree with the same contract.

  • Token complianceStates use role and tint tokens, not route-local hue maps.
  • Action clarityThe next member action is visible where a decision is needed.
  • Layout stabilityLoading, disabled, selected, pending, and RSVP states do not shift the surface.
  • Mobile readabilityPrimary actions appear before long bios, descriptions, or event notes.
  • AccessibilityState is not color alone; focus-visible and reduced motion are respected.
  • Editorial restraintDark/editorial treatment is reserved for entry moments, not dense workflows.
