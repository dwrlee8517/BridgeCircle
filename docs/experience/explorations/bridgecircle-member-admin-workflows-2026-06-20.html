North star

Turn verified school affiliation into useful support.

The member should feel safe asking. The helper should feel free to respond, pass, or set boundaries. The school should see an active, current, reachable community without turning the app into a donor CRM.

System model

One loop, two audiences

The same product loop should create member value and admin value. Members get useful help. Schools get current, engaged, reachable alumni.

Member inputs
S1

Profile and availability

Current role, city, school context, skills, topics, quick or ongoing availability.

S2

Questions and intent

Natural-language asks, standing asks, saved interests, event conditions later.

S3

Responses and boundaries

Accept, pass, decline reason, pause, capacity, active thread state.

Core loop

Verified circle -> relevant people -> respectful ask -> useful response -> stronger circle

Trust gate Invite, approval, org membership, field privacy.
Matching Search, helper preferences, capacity, shared context, visible evidence.
Conversation Ask review, accepted thread, direct message when friendship applies.
School pulse Events, announcements, profile freshness, admin next actions.
Admin outputs
S4

Activation dashboard

Invite-to-active, active members, help request response, freshness, RSVP activity.

S5

Recruit helpers, launch events, refresh stale profiles, resend invites, approve members.

S6

Clean data handoff

Export confirmed profile updates and engagement summaries back to existing systems.

Refinement prompts

Decisions to comment on first

These are the product choices most likely to affect whether the app feels easy and horizontal instead of formal and role-heavy.

D1

Should the public ask types be "quick" and "ongoing"?

Current implementation maps to advice and mentorship. The public mental model can be commitment-based without exposing hierarchy.

D2

Where should casual peer conversations live?

Inbox can include direct messages, accepted asks, and friend requests, but the groupings need to feel natural rather than bureaucratic.

D3

How strong should admin communication be in v1?

Events and announcements can increase engagement, but they should not turn the member app into a school bulletin board.

D4

What should count as a successful ask?

A reply, an accepted thread, a helpful pass, or a completed outcome may all matter. The product needs one primary success metric.

D5

How should BridgeCircle coexist with CRMs?

Best first version: import rosters, export confirmed profile deltas, and avoid donor/gift/household ownership.

D6

Should the School surface be a tab or a feed on Home?

A dedicated School tab is calm and scannable. A Home feed can increase discovery. The balance affects repeat use.
