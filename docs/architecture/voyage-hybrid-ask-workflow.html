BridgeCircle Ask Matching

Voyage hybrid workflow

The new Ask path keeps the current product promise: find trusted people who can help with a real question. It adds semantic recall through Voyage embeddings, keeps structured and lexical signals for precision, then uses Voyage reranking only after privacy and warm-network rules have narrowed the candidate pool.

ADR 0009 Model research Feature flag: ASK_MATCHING_PIPELINE=voyage_hybrid

Runtime Request

This is what happens when a member opens Ask with natural language, for example /ask?nl=I want advice from someone who moved from consulting to product.

1

Receive the ask

The Ask route passes the original natural-language intent into searchAlumniNL. Legacy remains the default unless the flag and Voyage key are both present.

2

Apply hard gates

Exclude impossible or unsafe matches first: wrong organization, inactive membership, private visibility, self-match, unavailable helper, or capacity conflict.

3

Retrieve in parallel

Pull candidates from structured rules, lexical matches, vector search over raw factual chunks, and vector search over semantic profile passages.

4

Merge and warm score

Dedupe by member, preserve evidence from each retrieval source, then apply trust and usefulness signals before reranking.

5

Voyage rerank

Send the top candidate evidence strings to rerank-2.5-lite. The reranker sees allowed evidence, not private profile rows or synthetic display copy.

6

Explain and hand off

Return 5-10 explained matches. Templated explanations are default; Haiku can polish final copy when ASK_MATCHING_EXPLANATIONS=haiku_polish.

Candidate Sources

Vector search improves recall, but it is not allowed to replace exact matching, privacy gates, or warm-network ranking.

Structured rules Known fields such as school, role, city, cohort, helper status, and explicit filters.
Lexical search Exact names, employers, titles, schools, topics, and profile words that must stay precise.
Raw chunk vectors Permission-safe profile facts embedded with voyage-4 at 1024 dimensions.
Semantic passage vectors Haiku-generated summaries from supplied facts only. Used for retrieval, never display evidence.

Profile Indexing

The background path prepares profile evidence before a member searches, so live Ask requests only need query embedding, retrieval, reranking, and explanation formatting.

Chunk families

Raw factual chunks
Generated from visible profile facts: current role, career history, education, skills, bio, mentoring topics, location, and helper preferences.
Semantic passages
Generated by Haiku from the same visible facts only. The prompt must forbid unsupported inference and preserve uncertainty.
Versioning
Store content_hash, synthetic_prompt_version, embedding_model, and embedding_dim for idempotent backfills and re-embedding.

Storage rules

  • Store chunks in profile_embedding_chunks with organization and membership scope.
  • Use visibility_tier to prevent private sections from becoming searchable.
  • Do not generate or embed self-only sections for Ask matching.
  • Allow friends chunks only when the viewer is already friends with the candidate.
  • Keep direct table reads blocked for normal users; retrieval runs through controlled server code.

Ranking Logic

The model stack does semantic ordering, but deterministic product signals still define what counts as a good BridgeCircle match.

Warm score

  • Shared school or organization context
  • Cohort proximity where useful
  • Shared city, major, industry, or path
  • Helper openness and response likelihood
  • Profile freshness and current availability

Reranker payload

  • Original ask as the query
  • Top 20-40 merged candidates
  • Privacy-redacted raw evidence strings
  • Retrieval-source hints for traceability
  • No private sections and no synthetic display evidence

Final result

  • 5-10 people ranked for actionability
  • Visible "why this match" evidence
  • Suggested first ask copy
  • CTA carries the original intent into /ask/new
  • Diagnostics log counts and fallback reason, not private text

Fallbacks

The new workflow is designed to fail closed and keep Ask usable while the index, provider key, and evals mature.

No Voyage key Use the existing legacy path: extraction, structured search, and Haiku reranking.
No chunks or vector failure Use structured plus lexical retrieval and warm scoring; skip vector candidates.
Reranker failure Return the merged warm-scored candidates with templated explanations.
Haiku polish failure Keep the same ranking and return templated explanations and default ask copy.

Practical default: start with voyage-4 embeddings and rerank-2.5-lite, keep ASK_MATCHING_PIPELINE=legacy as the fallback until migration, backfill, unit tests, and the existing Ask/People loop are verified.
