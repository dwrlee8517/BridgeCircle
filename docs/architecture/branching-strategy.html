Supabase Branching vs Two Projects

Visual explainer: how the BridgeCircle deploy workflow changes if we switch from bridgecircle-dev + bridgecircle (two separate projects) to a single project with a long-lived dev branch and PR-preview branches.

1. Mental model

Two ways to spell "dev and prod are separate." One uses two billing-distinct Supabase projects; the other uses one project with branches inside it (like git for your database).

Today

Two separate projects

bridgecircle-dev
Your dev DB. Has 9 personas.
Pointed at by local pnpm dev.
bridgecircle (prod)
Your prod DB. Empty until launch.
Pointed at by Railway deploys.

No relationship between them. Migrations are duplicated by your hands.

With branching

One project, many branches

bridgecircle (prod / "main")
The trunk. What real members hit.
dev (persistent)
Long-lived. Replaces bridgecircle-dev. Your daily local target.
pr-21 (preview)
Auto-created when PR opens. Auto-deleted on merge.
pr-22 (preview)
Each open PR gets its own.

Branches share parentage with prod. Migrations flow from PR → preview branch → merge → prod automatically.

Key shift Today, "dev" and "prod" are two databases that happen to share migration files. With branching, "dev" is a fork of "prod" — it inherits prod's migration history by construction, and merges that flow makes drift physically impossible.

2. Today's flow vs the branching flow

The same task — "ship a migration that adds a column" — viewed as a flow diagram.

Today (manual, two-project)

edit .sql
→
link to dev
→
db push
→
test locally
→
merge PR
→
link to prod
→
dry-run
→
db push
→
re-link to dev

9 steps   6 manual CLI invocations. The "I forgot to push to prod" failure mode lives in this flow.

With branching (git-driven)

edit .sql
→
git push
→
preview branch built
→
test on preview
→
merge PR
→
prod migrates

6 steps   0 manual CLI invocations. git push and merge are git ops you already do.

The actual win It's not "fewer steps" (that's surface). It's that the prod-push step becomes automatic. Today, prod-push is something I have to remember; tomorrow, it's a side effect of merging the PR.

3. What lives where

Branches are schema-level isolation. Some things flow from prod automatically; some don't. This table is what I'd want to remember during cutover.

ThingIn a new branch?How
Schema (tables, columns, indexes, RLS policies)YesReplays migrations from app/supabase/migrations/
Realtime publication membershipYesBecause we did it via migration (20260427233336_messages_realtime.sql)
Storage bucket definitionsYesDeclared in supabase/config.toml + bucket migration
Edge FunctionsYesIf declared in config.toml. We have none.
Production data (auth users, profiles, messages)NoBranches are data-less. Always.
Storage objects (uploaded files)NoJust bucket definitions, no objects.
Secrets (ANTHROPIC_API_KEY, RESEND_API_KEY, etc.)Per-branchSet separately on each persistent branch + prod
Seed data (our 9 personas)Manualseed-dev.ts isn't auto-run. Run it locally pointing at the dev branch.
Custom DB roles, dashboard-toggled settingsNoIf you click in the UI, branches don't see it. Migrations are the source of truth.
Why "data-less" is OK for us Today, dev and prod also don't share data — dev has personas, prod is empty. Branching keeps that property; it just makes it explicit. Post-launch we still wouldn't want prod data on dev (privacy, member trust). If we ever need a "test risky migration against real data" branch, we'd dump-and-restore manually. The pattern doesn't change.

4. A typical PR, side by side

Imagine I'm shipping the upcoming "admin approval queue" feature with a tiny migration that adds a reviewed_at column.

Today

What I do, in order

  1. Branch off main: git checkout -b admin-queue
  2. Write migration file
  3. Run pnpm dlx supabase db push on dev
  4. Run pnpm db:types
  5. Build feature, commit, push branch
  6. Open PR, test locally pointing at dev
  7. Merge to main
  8. Run pnpm dlx supabase link --project-ref <prod>
  9. Run pnpm dlx supabase db push --dry-run
  10. Run pnpm dlx supabase db push
  11. Run pnpm dlx supabase link --project-ref <dev>
With branching

What I do, in order

  1. Branch off main: git checkout -b admin-queue
  2. Write migration file
  3. (skipped — happens on push)
  4. Build feature, commit, push branch
  5. Supabase auto-creates a preview branch and applies migrations
  6. Open PR, test against the preview branch's URL
  7. Merge to main
  8. (skipped — auto-deploys to prod)
  9. Run pnpm db:types if I want fresh types locally

The 11-step list shrinks to ~7 effective steps. More importantly, the steps that disappear are the error-prone manual ones — the link/relink dance and the prod-push.

5. What concretely changes for me

Things that disappear

  • All supabase link --project-ref ... invocations
  • Manual db push to dev (auto-runs on PR push)
  • Manual db push to prod (auto-runs on merge to main)
  • The dry-run-then-push two-step before prod

Things that change shape

  • pnpm db:types now points at the persistent dev branch's project ref. Update once.
  • The CI status check on PRs becomes "Supabase Preview must succeed" — same idea as a build check, just different name.
  • "Check what migrations are pending on prod" becomes "look at the PR's status check" instead of "manually link and run migration list."

New requirements

  • Set every secret on each branch separately — ANTHROPIC_API_KEY, RESEND_API_KEY, SUPABASE_SECRET_KEY, etc. on the dev branch and on prod.
  • Decide where the seed runs (see below).
  • Add a GitHub branch protection rule on main that requires the Supabase Preview check to pass.

The seed-dev.ts wrinkle

Supabase's auto-seed only runs seed.sql files. Our seed is TypeScript using the JS client (because auth.admin.createUser is way nicer than raw SQL). Two options:

Option A: convert to seed.sql

Lose the ergonomic auth-creation. Translate every persona into raw SQL. Probably not worth it.

Option B: keep TS, run manually recommended

Run seed-dev.ts locally once after creating the persistent dev branch, pointing at the branch's connection string. Same as today, just one-time. Preview branches stay empty (you don't need 9 personas to test a migration).

6. Gotchas worth knowing

Working directory Our supabase/ lives at app/supabase/, not at the repo root. The GitHub integration's working directory field has to be set to app. If you leave it at ., branches will silently no-op on every PR.
Persistent-branch chicken-and-egg You can't add a [remotes.dev] block to config.toml until the persistent branch exists. Order: create branch → grab its project ref → write the config block → commit.
Rollback is destructive No "revert this migration" button. To undo: delete the preview branch and reopen the PR. Any data you typed into the branch is wiped. Treat preview branch state as throwaway.
Pre-cutover audit Anything I configured via the dashboard UI on the current dev project that isn't in a migration file won't carry over. Audit list:
  • Realtime publication members → in migration ✓
  • RLS policies → in migration ✓
  • Storage buckets → in migration ✓
  • Edge Functions → none ✓
  • Custom DB roles → none ✓
We're clean. The /lib discipline + migration-only schema work paid off here.
Pricing not in the docs The branching guide pages don't quote per-branch pricing. The configuration page links out to a separate pricing page. Verify what a persistent branch costs on Pro before decommissioning bridgecircle-dev. Don't trust any number a model quotes.

7. The cutover sequence

Realistic time estimate: 1–3 hours, mostly waiting and verifying. The actual clicking and editing is maybe 30 minutes.

  1. 1
    You
    Open Supabase pricing page. Confirm per-branch cost on Pro is acceptable. Stop here if not.
  2. 2
    You
    Dashboard → bridgecircle (prod) → Database → Branching → Enable. Connect the GitHub repo.
  3. 3
    You
    In the GitHub integration, set working directory to app (not .). Enable "Automatic branching" and "Deploy to production."
  4. 4
    You
    Create a persistent branch named dev via the dashboard. Wait for it to build (~1–2 minutes — runs the migration history).
  5. 5
    Me
    Add [remotes.dev] block to app/supabase/config.toml with the branch's project ref. Update pnpm db:types script if needed.
  6. 6
    You
    Copy secrets to both: persistent dev branch + prod. Settings → API → Project secrets. Each branch has its own.
  7. 7
    Me
    Run seed-dev.ts against the persistent dev branch's connection string. Verify 9 personas show up.
  8. 8
    Both
    Smoke test: sign in as Mark locally → search → friend request → DM → privacy settings. If all green, cutover succeeded.
  9. 9
    You
    GitHub repo → Settings → Branches → require "Supabase Preview" status check before merging to main.
  10. 10
    Me
    Update CLAUDE.md: remove the "branching as backlog" item, document the new workflow in Supabase Conventions.
  11. 11
    You (optional)
    After a week of green PRs, archive or delete the standalone bridgecircle-dev project to save the line item.

8. Should I do it?

Three lenses to look at the decision through.

Reasons to do it now

  • Pre-launch is the safest moment — prod has no real users to disturb.
  • Migrations already 100% in code (no dashboard drift), so cutover is mechanical.
  • You said you have time.
  • "I forgot to push to prod" risk goes away before real members exist.
  • The PR-preview infrastructure pays off if you ever bring on a 2nd engineer or alumni board reviewer.

Reasons to wait

  • Demo is in 4 weeks. Anything that touches infra burns demo-prep time.
  • Current setup has worked for 5 migrations in a row without drama.
  • One more failure mode to debug if something goes sideways at the worst time.
  • The wins are workflow improvements, not feature improvements — no demo-day visibility.
Recommendation If you have time and the per-branch pricing is reasonable, do it now. The workflow improvement compounds, and post-launch is a worse moment because real members exist. If pricing is unfriendly or you'd rather de-risk demo prep, the current two-project setup will keep working — branching is on the post-launch backlog regardless.
