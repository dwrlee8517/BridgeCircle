Identity & organization membership 6 tables

The foundation. Every other domain scopes through organization_id so multi-org never requires a destructive migration.

auth.users ───1—1───▶ users ───1—1───▶ base_profiles │ │ 1—N ▼ organization_memberships ───N—1───▶ organizations │ ▲ │ 1—1 │ 1—N ▼ │ organization_profiles invites
users
Shadow of auth.users. We don't put profile data here — Supabase owns that table.
id (FK auth.users.id), email, deleted_at, last_seen_at, created_at
organizations
Two rows at launch: Chadwick School, Chadwick International. Every other table scopes here.
id, slug, name, logo_url, created_at
organization_memberships
The join that does the real work. Approval queue lives in status.
id, user_id, organization_id, status, joined_at, approved_by, approved_at
status: pending | active | rejected | revoked
base_profiles
Reusable identity. One per user. Survives multi-org.
id, user_id, full_name, headline, current_employer, current_title, city, university, major, linkedin_url, avatar_url
organization_profiles
Org-specific overlay. UI merges this with base_profiles at launch — split is invisible until org #2.
id, membership_id, graduation_year, bio, mentoring_topics, open_to_mentor
invites
The promise to join. Token validation lives here, before any user exists.
id, organization_id, email, token, status, expires_at, sent_by, accepted_by, full_name?, graduation_year?
status: pending | accepted | expired | revoked

Why split base_profile / organization_profile?

UI shows one profile at launch. But when Chadwick International onboards as org #2, an existing user adds an organization_profile row without rebuilding identity. Without the split, multi-org needs a destructive migration.

Why invites separate from memberships?

Invite is a promise; membership is the fact. Token validation works before signup, when no user row exists. A revoked invite shouldn't disturb membership history.

Mentorship 4 tables — the core launch loop

Request → accept → thread → messages. The request and the conversation are deliberately separate tables.

mentorship_preferences (per organization_membership) │ │ governs who appears as "open to mentor" ▼ mentorship_requests ──(status=accepted)──▶ mentorship_threads │ 1—N ▼ messages (thread_type='mentorship')
mentorship_preferences
Per organization_membership, not per user — open-to-mentor at one org doesn't mean open at another.
id, membership_id, is_open, topics[], screening_prompt, max_active_mentees, max_pending_requests, paused_at
mentorship_requests
The application. Closes once decided. Carries the screening answer.
id, mentor_id, mentee_id, organization_id, status, reason, help_needed, background, screening_answer, responded_at
status: pending | accepted | declined | expired
mentorship_threads
Created on accept. Outlives the request — a future "re-request" creates a new thread.
id, request_id, mentor_id, mentee_id, status, last_message_at
status: active | archived
messages
Polymorphic by enum. One inbox query covers mentorship + DM threads.
id, thread_id, thread_type, sender_id, body, read_at, created_at
thread_type: mentorship | direct

Why thread separate from request?

Different lifecycle. Requests close once decided; threads stay open until archived. A future "re-request the same mentor" feature creates a new request and new thread, leaving history intact.

Why one messages table for both DM and mentorship?

Inbox = "all unread messages for user X." One scan instead of two unioned scans. Notification fan-out and moderation also become single-surface. The thread_type enum is the only branching cost.

Events 2 tables

Minimum surface. No waitlist, no maybe, no plus-ones. Add only if real demand surfaces.

organizations ───1—N───▶ events ───1—N───▶ event_rsvps ───N—1───▶ users
events
Title, time, place, description. published_at separates drafts from live.
id, organization_id, created_by, title, description, location, starts_at, ends_at, published_at
event_rsvps
One row per (event, user). Going or not — that's the whole vocabulary.
id, event_id, user_id, status, responded_at
status: going | not_going

Admin & audit 2 tables

Per-org role scoping. Audit writes from Day 1 even though we don't build a viewer until later.

admin_role_assignments
Per-org elevation above member. Founder is super_admin of both orgs independently.
id, user_id, organization_id, role, granted_by, granted_at
role: super_admin | admin | event_moderator | ambassador
audit_log
Write-only at launch. The first time you need it, it's too late to add — so it ships now.
id, actor_id, organization_id, action, target_type, target_id, payload (jsonb), created_at

Social schema only 3 tables

Friend requests and DMs land in week 3. Schema ships now so foreign keys resolve and week 3 becomes UI work, not a migration.

friend_requests ──(accept)──▶ friendships ──(first DM)──▶ direct_message_threads ───▶ messages (one row per pair, (created lazily) user_a_id < user_b_id)
friend_requests
Pending requests with optional intro message. Same request-then-relationship pattern as mentorship.
id, requester_id, recipient_id, intro_message, status, responded_at
status: pending | accepted | declined
friendships
One row per friendship. CHECK constraint enforces user_a_id < user_b_id — no duplicates.
id, user_a_id, user_b_id, became_friends_at CHECK (user_a_id < user_b_id)
direct_message_threads
One per friend pair. Created lazily on first DM, not at friendship time.
id, user_a_id, user_b_id, last_message_at CHECK (user_a_id < user_b_id)

Why store friendships canonically (one row, ordered ids)?

Two-row representation ("X is friends with Y" + "Y is friends with X") doubles writes and creates consistency risk. One row with user_a_id < user_b_id makes "are they friends?" a single PK lookup and makes the unique index trivial.

System schema only 4 tables

Announcements, in-app notifications, profile freshness prompts, and saved searches. UIs come in week 3+ but the rows can start being written sooner if needed.

announcements
Org-scoped, admin-authored. Week 3 surface.
id, organization_id, author_id, title, body, published_at
notifications
Per-user delivery log. Email-only at launch, but rows still get written so the in-app tray works on day one of week 3.
id, user_id, type, payload (jsonb), read_at, created_at
profile_refresh_prompts
Per organization_membership. Drives the 6–12 month freshness loop.
id, membership_id, due_at, last_confirmed_at, dismissed_at
saved_searches
Per-user. Stores filter blob and optional notify cadence for "alert me when matches appear."
id, user_id, name, filters (jsonb), notify_cadence, last_notified_at

Lifecycle flows

The three paths every member takes through the schema.

Onboarding — invite to active member

1. Admin uploads CSV ───▶ insert invites (status=pending) 2. Resend sends token email ───▶ audit_log: invite.sent 3. User clicks /join?token=xyz 4. Server validates token ───▶ match against invites 5. User signs up via Supabase Auth ───▶ trigger creates users row 6. Token auto-approves ───▶ insert organization_memberships (status=active) ───▶ invite.status = accepted 7. Profile setup screen ───▶ insert base_profiles + organization_profiles 8. audit_log: member.joined

Mentorship request — the repeat loop

1. Mentee searches ───▶ filter by mentorship_preferences.is_open 2. Mentee opens profile ───▶ request composer 3. Submit ───▶ insert mentorship_requests (status=pending) 4. Resend emails mentor ───▶ insert notifications row 5. Mentor accepts ───▶ request.status = accepted ───▶ insert mentorship_threads ───▶ insert system messages row 6. Both inboxes show thread 7. audit_log: mentorship.accepted

Mentor inactivity sweep — Railway worker, daily

1. SELECT mentorship_preferences WHERE is_open=true AND paused_at IS NULL 2. For each: any mentorship_requests older than 14 days, status=pending? 3. If yes ───▶ mentorship_preferences.paused_at = now() 4. Search rankings drop "paused while away" mentors 5. On next mentor login ───▶ unpause prompt ───▶ paused_at = null

Cross-cutting decisions

Every scoped table carries organization_id

Mentorship requests, events, admin roles, profile overlays, audit rows — all scope by org. RLS becomes "is the requester an active member of this organization_id?" — no multi-hop joins.

Status enums, not boolean flags

Booleans drift into illegal combinations (is_pending=true AND is_accepted=true). Enums make illegal states unrepresentable.

audit_log writes from Day 1

The first time you need it (mentor reports unauthorized signup, admin says "I never approved that"), it's too late to add. Write rows from the first admin action — no gaps.

What's deliberately not here

No meetup_proposals, no mentorship_response table (response is a state on the request), no profile_visibility_setting table (defaults hardcoded in app code), no ambassador workflow tables. Each is a deliberate scope cut.
