# 0009 — Hybrid Ask matching

- **Status:** accepted
- **Date:** 2026-06-08
- **Decider:** Richard

## Context

ADR 0006 chose entity extraction into structured search, with no vector search,
because Phase 1 had a small member base and the first NL search box was mostly
directory exploration.

Ask is now the primary question-led member surface. The user starts with a real
question and expects BridgeCircle to find people who fit that need, not just
people whose profile contains extracted entities. Entity extraction plus
structured search is interpretable and cheap, but it can miss helpers whose
career history, skills, bio, or mentoring topics are semantically relevant
without exact filter overlap.

Current production code still implements the ADR 0006-style baseline:
structured/entity retrieval plus Claude Haiku reranking. This ADR defines the
accepted target architecture for the next Ask matching upgrade.

The embedding and reranker provider analysis in
[Ask matching model research](../architecture/ask-matching-model-research.md)
supports a hosted Voyage-first implementation path: Voyage embeddings for
semantic retrieval, Voyage reranking for the default page-load ranker, and
Claude Haiku reserved for final explanation and ask-copy support rather than
candidate-pool scoring.

## Decision

Default Ask results should use **hybrid retrieval + warm-network scoring +
dedicated reranking**, with LLM support reserved for query understanding,
explanation polish, and ask-copy generation.

Profile embeddings should include both:

- **raw factual chunks** from permission-safe profile sections, such as current
  role, career history, education history, skills, bio, and mentoring topics
- **semantic profile passages** generated by an LLM from those same
  permission-safe facts, such as career-path summaries and help-topic summaries

Semantic profile passages are retrieval aids, not source-of-truth evidence.
They must be strictly fact-grounded, versioned by prompt/model/content hash, and
regenerated when the underlying visible profile facts change. Match
explanations must cite raw visible evidence, not synthetic prose.

The target pipeline:

1. Apply hard gates first: authenticated viewer, shared active organization,
   profile privacy, helper availability, mentorship capacity, and self-exclusion.
2. Retrieve candidates in parallel:
   - structured/rules search for explicit filters and known profile fields
   - lexical search for exact names, titles, employers, schools, topics, and
     profile text
   - vector semantic search over permission-safe raw profile chunks
   - vector semantic search over permission-safe semantic profile passages
3. Merge and dedupe candidates by `user_id`, preserving evidence from each
   retrieval source.
4. Score the merged pool with warm-network signals: helper openness, shared
   school, cohort proximity, shared city, major, relationship path, profile
   freshness, helper load, and response likelihood where available.
5. Rerank the top 20-40 candidates against the original ask with a dedicated
   reranker, using only privacy-allowed evidence. The first implementation
   candidate is Voyage `rerank-2.5-lite`; evaluate `rerank-2.5` if local Ask
   fixtures show meaningful quality misses.
6. Return 5-10 explained matches on `/ask?nl=...`, with "Why this match",
   suggested first ask copy, and a CTA that carries the original ask into
   `/ask/new?...&intent=...`. Explanations may be templated from raw evidence
   immediately; an LLM may polish final explanations and suggested ask copy for
   the final 5-10 candidates.

Fallbacks are part of the decision:

- If vector search is unavailable, use structured + lexical retrieval.
- If semantic profile passage generation is unavailable, embed raw factual
  chunks only.
- If the dedicated reranker is unavailable, return the merged scored candidates
  with templated explanations.
- If the LLM is unavailable, keep the same ranked candidates and return
  templated explanations and ask-copy defaults.
- If retrieval returns weak evidence, Ask may offer query refinement or a deeper
  async matching pass rather than showing noisy confidence.

People remains the broad directory/search-and-filter surface. It can share
lower-level retrieval helpers, but it should not inherit agentic Ask behavior by
default.

## Consequences

- **+** Better recall for real asks that do not reduce cleanly to entity filters.
- **+** Keeps exact-name and structured filter behavior strong by retaining
  lexical and rules-based retrieval.
- **+** Preserves BridgeCircle's warm-network thesis because trust, availability,
  and relationship signals remain first-class ranking inputs.
- **+** Improves semantic recall for narrative asks by embedding generated
  career-path and help-topic passages rather than only raw concatenated fields.
- **+** Keeps explanations human-readable by preserving raw retrieval evidence
  and using LLM-generated text only as final explanation/copy support.
- **+** Lowers default page-load latency and cost by using a dedicated reranker
  instead of an LLM as the main reranking step.
- **−** Requires new infrastructure: `pgvector`, profile embedding storage,
  embedding backfill, profile-update re-embedding, and vector index maintenance.
- **−** Requires privacy discipline. Private profile sections must not become
  globally searchable through embeddings; chunks need visibility metadata or
  must be limited to org-visible/helper-visible content.
- **−** Adds synthetic-passage generation risk. The generation prompt must
  forbid unsupported inferences, and generated passages must never be cited as
  user-facing evidence.
- **−** Adds embedding-version complexity. Raw chunks and semantic passages need
  content hashes, generator model/prompt versions, and re-embedding on profile
  updates.
- **−** Adds latency. Default Ask target is roughly 2 seconds with dedicated
  rerank and up to 2-5 seconds when optional LLM explanation polish is used;
  fallback retrieval should remain under roughly 1 second.
- **−** Requires evaluation. The team needs representative ask fixtures,
  expected-good matches, unacceptable matches, source-level latency metrics, and
  click/send outcome tracking.

## Alternatives considered

- **Keep entity extraction only.** Cheapest and already implemented, but misses
  semantically relevant helpers when the ask is not reducible to a few fields.
- **Vector-only search.** Good semantic recall but weak for hard constraints,
  exact names, availability, capacity, and trust signals.
- **Raw concatenated embeddings only.** Simpler and lower-risk, but likely worse
  for narrative asks like career transitions, founder paths, and "someone who
  has walked this path" phrasing.
- **Use synthetic profile passages as display evidence.** Rejected. Generated
  passages can improve retrieval, but user-facing explanations must cite raw
  visible profile facts to avoid hallucination and privacy leakage.
- **LLM reranking as the default page-load ranker.** Flexible and currently
  implemented in the baseline, but slower, more expensive, and less
  deterministic than a dedicated reranker. Keep LLMs for final explanation
  polish, ask-copy support, and possible async deep matching.
- **Full agentic matching by default.** Potentially high quality, but too slow
  and opaque for the default page load. Keep it as a possible async deep-match
  mode after the hybrid default is working.
- **Human-mediated matching.** Highest trust for high-stakes asks, but not the
  default self-serve product loop.
